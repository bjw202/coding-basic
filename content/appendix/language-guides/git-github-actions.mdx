## 1. Git 기초

Git은 "분산 버전 관리 시스템"입니다.

**왜 필요한가?**
- "최종.zip", "진짜최종.zip" 지옥에서 탈출
- 누가 언제 무엇을 변경했는지 추적
- 실수해도 이전 상태로 복구 가능
- 팀원들과 충돌 없이 협업

### 설치 및 초기 설정

```bash
# macOS
brew install git

# 사용자 정보 설정 (필수!)
git config --global user.name "홍길동"
git config --global user.email "hong@example.com"

# 기본 브랜치명 설정
git config --global init.defaultBranch main
```

## 2. Git의 3가지 영역

```
Working Directory    Staging Area      Repository
(작업 디렉토리)       (스테이징)        (저장소)

[파일 수정]  ──▶  [git add]  ──▶  [git commit]

실제 작업하는        커밋 준비         변경 이력
폴더                영역              저장소
```

---

## 3. Git 필수 명령어

```bash
# 저장소 생성
git init
git clone https://github.com/user/repo.git

# 상태 확인
git status
git log --oneline

# 변경사항 스테이징 및 커밋
git add .
git add src/api.js
git commit -m "feat: 로그인 API 추가"

# 원격 저장소
git remote add origin https://github.com/user/repo.git
git push origin main
git pull origin main
```

---

## 4. 브랜치 전략

```bash
# 브랜치 생성 및 이동
git branch feature/login
git checkout feature/login
git checkout -b feature/signup  # 생성 + 이동

# 브랜치 목록
git branch -a

# 브랜치 병합
git checkout main
git merge feature/login

# 브랜치 삭제
git branch -d feature/login
```

### 실무 브랜치 전략

```
main          ← 프로덕션 배포
develop       ← 개발 통합
feature/*     ← 기능 개발
hotfix/*      ← 긴급 수정
```

---

## 5. 커밋 메시지 컨벤션

```
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 스타일 변경 (기능 변경 없음)
refactor: 코드 리팩토링
test: 테스트 추가/수정
chore: 빌드 도구 변경, 설정 변경
```

예시:
```
feat: 사용자 로그인 API 추가
fix: 비밀번호 유효성 검사 오류 수정
docs: README에 API 사용법 추가
```

## 6. GitHub 협업 워크플로우

1. 이슈 생성 (무엇을 할지 정의)
2. 브랜치 생성 (`feature/issue-번호-기능명`)
3. 개발 및 커밋
4. PR(Pull Request) 생성
5. 코드 리뷰
6. 머지 (Merge)
7. 브랜치 삭제

## 7. GitHub Actions CI/CD

GitHub Actions는 코드 변경 시 자동으로 테스트, 빌드, 배포를 실행합니다.

### 기본 워크플로우 파일

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: 패키지 설치
        run: npm install

      - name: 테스트 실행
        run: npm test

      - name: 빌드
        run: npm run build
```

### 자동 배포 워크플로우

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Vercel 배포
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.ORG_ID }}
          vercel-project-id: ${{ secrets.PROJECT_ID }}
          vercel-args: '--prod'
```

> **Session 12 연결**: 12회차에서 배우는 CI/CD 파이프라인이 바로 이 GitHub Actions를 기반으로 합니다. `git push`를 하면 자동으로 테스트가 실행되고, 통과하면 서버에 자동 배포됩니다.

## 핵심 포인트 정리

- Git은 처음엔 외워서 쓰고, 나중엔 원리를 이해하게 됩니다. `add → commit → push` 흐름부터 시작하세요
- 브랜치 전략은 팀 규모에 맞게 단순하게 유지하는 것이 좋습니다
- 커밋 메시지는 무엇을 했는지가 아니라 왜 했는지를 적으면 나중에 이력을 읽기 쉽습니다
- GitHub Actions는 "무료 자동화 서버"입니다. PR이 올라오면 테스트를 자동으로 실행하는 것만으로도 큰 가치가 있습니다
