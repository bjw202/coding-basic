# 규모별 CLAUDE.md

## 학습 목표

이 자료를 마치면 다음을 할 수 있습니다:
- 조직 규모에 따라 AI 코딩 규칙의 강도를 다르게 설계할 수 있습니다
- 각 규모별 프롬프트 템플릿을 실무에 바로 적용할 수 있습니다
- "같은 철학, 다른 강도"의 원칙을 이해할 수 있습니다

공통 철학: **AI는 빠른 손, 책임은 사람**

---

## 1인 SaaS용

**상황**: 혼자 개발·운영·배포까지 담당
**목표**: 속도 최우선 + 치명적 사고만 방지

### 핵심 원칙 (최소 규칙)

```markdown
## 핵심 원칙

- 범위 지정 없는 수정 금지
- API 응답 스키마 변경 금지
- 인증/결제 로직은 반드시 직접 리뷰
```

### 허용 범위

```markdown
## 허용 범위

- UI/컴포넌트 수정 자유
- 내부 리팩토링은 테스트 통과 시 허용
```

### 전용 규칙

```markdown
## 1인 SaaS 전용 규칙

- 테스트 없을 경우라도 롤백 가능한 변경만 허용
- 배포 전 최소 체크:
  - 서버 정상 기동
  - 핵심 API 1~2개 직접 호출
```

### 프롬프트 템플릿

```
[범위] src만 수정
[목적] 특정 기능 개선
[제약] API 응답 스키마 유지, 인증 로직 변경 금지
[검증] 서버 재시작 후 핵심 API 수동 테스트
```

---

## 스타트업용

**상황**: 3~10명 규모, 빠른 기능 출시 + 잦은 변경
**목표**: 속도와 안정성의 균형

### 핵심 원칙

```markdown
## 핵심 원칙

- 1 PR = 1 기능
- AI 변경 PR은 [AI] 라벨 필수
- 테스트 없는 로직 변경 금지
```

### 역할 분리

```markdown
## 역할 분리

- AI: 구현/리팩토링
- 사람: 설계 판단/머지 승인
```

### 전용 규칙

```markdown
## 스타트업 전용 규칙

- API 계약 변경 시 반드시 문서화
- 장애 가능 변경은 feature flag 사용 권장
```

### 프롬프트 템플릿

```
[범위] src/api/orders
[목적] 주문 중복 생성 방지
[제약] 기존 API 계약 유지
[검증] 테스트 추가 + 기존 테스트 통과
```

---

## 엔터프라이즈용

**상황**: 다수 팀, 대규모 사용자, 장애 비용 큼
**목표**: 안정성·추적성·책임 명확화

### 핵심 원칙 (강제)

```markdown
## 핵심 원칙

- 범위 지정 없는 프롬프트 즉시 거부
- AI 단독 커밋/머지 금지
- 모든 변경은 감사 가능해야 함
```

### 변경 통제

```markdown
## 변경 통제

- API/DB/보안 변경은 사전 승인 필수
- 롤백 플랜 없는 PR 병합 금지
```

### 전용 규칙

```markdown
## 엔터프라이즈 전용 규칙

- 모든 AI PR에 영향도 분석 포함
- 로그/모니터링 변경은 SRE 리뷰 필수
```

### 프롬프트 템플릿

```
[범위] src/payments (read-only: auth, infra)
[목적] 결제 실패 케이스 보완
[제약] API 계약·보안 로직 변경 금지
[검증] 테스트 + 스테이징 검증 후 배포
```

---

## 규모별 규칙 강도 요약

| 항목 | 1인 SaaS | 스타트업 | 엔터프라이즈 |
|------|---------|---------|------------|
| 속도 | 매우 높음 | 높음 | 중간 |
| 규칙 강도 | 낮음 | 중간 | 매우 높음 |
| 테스트 강제 | 선택 | 필수 | 필수+확장 |
| 승인 절차 | 없음 | 1인 리뷰 | 다중 승인 |
| 롤백 요구 | 권장 | 필수 | 필수+문서 |

---

## 핵심 포인트 정리

- 규모가 커질수록 AI 규칙은 느슨함에서 강제 규율로 진화해야 합니다
- 같은 CLAUDE.md를 쓰는 것이 아니라, 같은 철학을 다른 강도로 적용하세요
- AI 코딩의 성숙도는 코드가 아니라 규칙의 수준에서 갈립니다
- 혼자 시작해도 나중에 팀이 될 것을 염두에 두고 규칙을 설계하세요
