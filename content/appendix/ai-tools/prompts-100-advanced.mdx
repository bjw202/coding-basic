# 고급 프롬프트 100선

## 학습 목표

이 자료를 마치면 다음을 할 수 있습니다:
- 아키텍처 검토와 대규모 리팩토링을 AI와 함께 안전하게 진행할 수 있습니다
- 보안, 성능, 안정성을 고려한 운영 수준의 프롬프트를 작성할 수 있습니다
- 장애 대응과 배포 안전 점검을 AI 도움으로 수행할 수 있습니다

원칙: **범위 제한 · 변경 최소 · 운영 안정성 최우선**

---

## 1. 아키텍처 / 설계 검토 (1~10)

```
현재 아키텍처의 핵심 책임 분리를 설명해줘
이 구조에서 확장성 병목이 될 부분을 찾아줘
모놀리식으로 유지하는 게 나은지 분리할지 평가해줘
이 프로젝트에 레이어드 아키텍처가 적절한지 판단해줘
도메인 경계가 잘못된 부분이 있는지 지적해줘
기술 부채로 보이는 부분을 우선순위로 정리해줘
트래픽 증가 시 가장 먼저 터질 가능성이 있는 지점을 알려줘
현재 구조에서 테스트가 어려운 이유를 분석해줘
운영 관점에서 위험한 의존성을 찾아줘
지금 상태로 1년 운영했을 때 문제를 예측해줘
```

---

## 2. 대규모 리팩토링 (11~20)

```
기능은 유지하고 구조만 단계적으로 개선해줘
이 변경을 여러 PR로 나누는 전략을 제안해줘
레거시 코드와 신규 코드의 경계를 명확히 해줘
공통 모듈로 분리 가능한 부분을 식별해줘
순환 의존성이 있는지 검사해줘
리팩토링 중 반드시 테스트로 보호해야 할 영역을 알려줘
설정(config)과 비즈니스 로직을 분리해줘
이 코드에서 SOLID 원칙 위반 사례를 찾아줘
리팩토링 후 예상되는 리스크를 나열해줘
롤백 전략까지 고려해서 구조 개선안을 제시해줘
```

---

## 3. 성능 최적화 (21~30)

```
실제 병목이 될 가능성이 높은 코드를 찾아줘
불필요한 DB 호출을 줄일 수 있는 방법을 제안해줘
캐시 적용이 효과적인 지점을 알려줘
N+1 문제가 발생할 수 있는 부분을 찾아줘
동기/비동기 처리 구조를 재검토해줘
메모리 사용량이 증가할 수 있는 패턴을 찾아줘
GC 부담을 줄이기 위한 개선점을 제안해줘
로깅 때문에 성능 저하가 생길 수 있는 부분을 알려줘
트래픽 급증 시 성능 열화 시나리오를 만들어줘
성능 테스트 전략을 제안해줘
```

---

## 4. 안정성 / 장애 대응 (31~40)

```
이 서비스에서 단일 장애 지점(SPOF)을 찾아줘
예외 처리가 누락된 경로를 모두 찾아줘
장애 발생 시 로그만으로 원인 추적이 가능한지 평가해줘
재시도 로직이 필요한 구간을 식별해줘
타임아웃 설정이 필요한 외부 연동 지점을 찾아줘
장애 전파를 막기 위한 방어 코드를 추가해줘
서킷 브레이커 패턴 적용 여부를 검토해줘
운영 중 발생할 수 있는 치명적 장애 시나리오를 작성해줘
장애 대응 매뉴얼 초안을 만들어줘
무중단 배포 시 위험 포인트를 알려줘
```

---

## 5. 보안 / 권한 (41~50)

```
인증/인가 로직에서 취약할 수 있는 부분을 찾아줘
민감 정보가 로그에 남을 가능성이 있는지 검사해줘
환경 변수/시크릿 관리 상태를 점검해줘
권한 상승 취약점이 생길 수 있는 코드를 찾아줘
입력값 검증이 부족한 부분을 모두 나열해줘
CSRF/XSS 관점에서 위험 요소를 분석해줘
API 오용으로 인한 데이터 노출 가능성을 점검해줘
보안 사고 발생 시 영향 범위를 정리해줘
보안 로그를 어떻게 남기는 게 좋은지 제안해줘
최소 권한 원칙이 지켜지고 있는지 평가해줘
```

---

## 6. 운영 / 배포 (51~60)

```
현재 배포 방식의 위험 요소를 분석해줘
롤백이 안전하게 가능한지 점검해줘
환경별(dev/stage/prod) 설정 분리가 잘 됐는지 봐줘
배포 중 장애가 발생할 수 있는 지점을 찾아줘
무중단 배포 전략을 제안해줘
배포 자동화에서 빠진 안전 장치를 알려줘
마이그레이션 시 주의할 점을 정리해줘
배포 전 체크리스트를 만들어줘
배포 후 반드시 확인해야 할 항목을 정리해줘
새 운영자가 와도 이해할 수 있는 배포 설명을 써줘
```

---

## 7. 데이터 / DB (61~70)

```
이 스키마에서 정규화가 필요한 부분을 찾아줘
인덱스가 필요한 컬럼을 추천해줘
대량 데이터에서 느려질 쿼리를 예측해줘
트랜잭션 경계가 올바른지 검토해줘
데이터 정합성이 깨질 수 있는 시나리오를 작성해줘
소프트 삭제/하드 삭제 전략을 평가해줘
데이터 마이그레이션 리스크를 분석해줘
백업/복구 전략을 점검해줘
장애 시 데이터 손실 가능성을 평가해줘
장기 운영 시 데이터 관리 전략을 제안해줘
```

---

## 8. 테스트 / 품질 (71~80)

```
현재 테스트 커버리지의 맹점을 찾아줘
통합 테스트가 필요한 핵심 흐름을 정의해줘
운영 장애를 재현할 테스트 시나리오를 만들어줘
테스트가 과도하게 느려질 수 있는 원인을 찾아줘
flaky 테스트 가능성을 분석해줘
테스트 없이 변경하면 위험한 코드를 식별해줘
QA 관점에서 놓치기 쉬운 케이스를 알려줘
회귀 테스트 전략을 정리해줘
테스트 자동화 성숙도를 평가해줘
품질 지표로 무엇을 봐야 하는지 제안해줘
```

---

## 9. 코드 리뷰 / 협업 (81~90)

```
이 변경에서 리뷰어가 집중해야 할 부분을 정리해줘
리뷰 기준으로 삼을 체크리스트를 만들어줘
코드 스타일 불일치 지점을 찾아줘
주니어가 오해할 수 있는 코드를 지적해줘
주석이 필요한 부분과 불필요한 부분을 구분해줘
PR을 더 작게 쪼갤 수 있는 방법을 제안해줘
리뷰 피드백을 반영한 개선안을 만들어줘
협업 시 충돌이 잦을 영역을 예측해줘
문서화가 필요한 부분을 정리해줘
기술 결정 기록(ADR) 초안을 작성해줘
```

---

## 10. 운영 마스터 제어 프롬프트 (91~100)

이 프롬프트들은 다른 프롬프트와 **반드시 함께** 사용합니다:

```
변경은 최소한으로, 안정성을 최우선으로 고려해줘
운영 중단 가능성이 있는 변경은 하지 말아줘
롤백 가능한 방식으로만 수정해줘
기존 API 계약을 절대 깨지 말아줘
로그/모니터링을 함께 보완해줘
장애 대응을 고려해서 코드를 수정해줘
프로덕션 환경 기준으로만 판단해줘
위험하면 반드시 경고해줘
수정 전에 예상 리스크를 먼저 나열해줘
내가 확인해야 할 포인트를 마지막에 정리해줘
```

---

## 고급 사용 팁

- 항상 범위 제한 문장과 함께 사용합니다
- 50개 이상 변경이 예상되면 반드시 PR을 분리합니다
- Claude 수정 → 사람 검증 → 배포 순서를 절대 바꾸지 않습니다

---

## 핵심 포인트 정리

- 이 문서는 운영을 맡은 개발자의 안전벨트입니다
- 빠르게 짜되, 절대 무너지지 않게 — 이것이 시니어의 기준입니다
- 1~10번 프롬프트들은 개발보다 설계 단계에 더 유용합니다
- 91~100번 제어 프롬프트는 특히 보안, 배포, DB 변경 시 반드시 함께 사용하세요
