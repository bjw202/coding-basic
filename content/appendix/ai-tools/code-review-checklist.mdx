# 코드리뷰 체크리스트

## 학습 목표

이 자료를 마치면 다음을 할 수 있습니다:
- AI가 만든 PR을 체계적으로 검증할 수 있습니다
- 팀에서 바로 사용할 수 있는 PR 템플릿을 설정할 수 있습니다
- AI 변경에서 자주 나타나는 레드플래그를 즉시 감지할 수 있습니다

---

## PR 제목 규칙

AI가 작성한 코드임을 명시합니다:

```
[AI] feat: 주문 중복 생성 방지 로직 추가
[AI] fix: 로그인 API null 방어 처리
[AI] refactor: UserService 가독성 개선
```

---

## PR 설명 템플릿 (.github/pull_request_template.md)

```markdown
## 1) 변경 요약
<!-- 무엇을 바꿨는지 3줄 이내로 작성 -->
<!-- 핵심 파일/모듈 나열 -->

## 2) 변경 이유
<!-- 왜 필요한지 -->
<!-- 기존 문제(버그/성능/운영 이슈) -->

## 3) 영향 범위
<!-- 영향 받는 기능/사용자/모듈 -->
<!-- API 계약(요청/응답) 변경 여부 -->

## 4) 검증 방법
<!-- 실행한 테스트/검증 단계 -->
<!-- 재현/확인 명령어 -->

## 5) 롤백 전략
<!-- 롤백 가능 여부 -->
<!-- 롤백 시 체크 포인트 -->
```

## AI 코드 리뷰 체크리스트 (9개 항목)

### A. 범위/의도 확인

- [ ] PR 목적이 한 가지인가? (1 PR = 1 목적)
- [ ] 변경 범위가 과도하지 않은가? (파일 수/라인 수)
- [ ] "깔끔하게/더 좋게" 같은 추상 이유가 아닌가?

### B. API/계약 안정성

- [ ] 기존 API 응답/요청 스키마가 유지되는가?
- [ ] 변경이 있다면 버전업/마이그레이션 계획이 있는가?
- [ ] 프론트/클라이언트 영향이 문서화 되었는가?

### C. 데이터/DB 안정성

- [ ] 쿼리/스키마 변경이 있는가?
- [ ] 트랜잭션 경계가 적절한가?
- [ ] 인덱스/성능 영향 검토가 포함되었는가?

### D. 에러 처리/엣지 케이스

- [ ] null/undefined 방어가 있는가?
- [ ] 외부 연동(HTTP/DB/Queue) 타임아웃/재시도 고려가 있는가?
- [ ] 사용자 입력 검증이 있는가?

### E. 보안/권한

- [ ] 인증/인가 로직이 변경되었는가?
- [ ] 민감 정보(토큰/비번/개인정보)가 로그에 남지 않는가?
- [ ] 권한 우회 가능성이 없는가?

### F. 로깅/관측성(Observability)

- [ ] 운영에 필요한 로그가 제거되지 않았는가?
- [ ] 에러 로그에 requestId/핵심 정보가 포함되는가?
- [ ] 메트릭/트레이싱이 필요한 변경인가?

### G. 테스트

- [ ] 최소 1개 이상 테스트가 추가/수정되었는가?
- [ ] 회귀 테스트(이전 버그 재현)가 포함되었는가?
- [ ] flaky 가능성(시간/순서 의존)이 없는가?

### H. 성능

- [ ] O(n²) 같은 위험한 루프가 추가되지 않았는가?
- [ ] N+1, 불필요 호출/폴링이 생기지 않았는가?
- [ ] 캐시/커넥션풀 등 운영 설정 영향이 있는가?

### I. 배포/운영

- [ ] config/env 변경이 있는가?
- [ ] 마이그레이션/배포 순서가 적혀 있는가?
- [ ] 롤백 시 안전한가?

## 리뷰어용 빠른 확인 명령

변경 확인:

```bash
git fetch
git diff origin/main...HEAD
```

테스트:

```bash
npm test
npm run build
```

API 스모크 테스트:

```bash
curl -i http://localhost:3000/health
```

## AI 변경에서 자주 터지는 레드플래그

다음 패턴이 보이면 즉시 확인하세요:

- 응답 키 이름 변경 (`items` → `data` 등)
- 로깅 제거
- 예외 처리 누락
- "성능 개선"이라고 하면서 부하 조건 미정
- 보안/인증 로직 단순화

---

## 핵심 포인트 정리

- AI 코드 리뷰는 '코드 품질'이 아니라 운영 사고를 막는 절차입니다
- PR 템플릿은 팀의 안전벨트입니다. `.github/pull_request_template.md`에 바로 복붙하세요
- 레드플래그 5가지를 외우면 검토 시간이 절반으로 줄어듭니다
- 모든 체크리스트 항목에 "아니오"가 있을 때만 PR을 머지하지 마세요. 판단은 사람이 합니다
